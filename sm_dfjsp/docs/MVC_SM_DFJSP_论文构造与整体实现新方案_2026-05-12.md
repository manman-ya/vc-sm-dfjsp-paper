> Current project rule: cross-chain penalty adds fixed cross-chain cost only; `cross_chain_cost_rate` is kept for compatibility and is always 0.

# 面向多服务价值链协同的 MVC-SM-DFJSP 论文构造与整体实现新方案

创建日期：2026-05-12  
依据文档：

- `D:/code/recode/vc-sm_dfjsp/sm_dfjsp/docs/ChatGPT-构造FJSP实例扩展.md`
- `D:/code/recode/vc-sm_dfjsp/sm_dfjsp/docs/MVC_SM_DFJSP_整体实现方案_2026-05-11.md`

建议题目：

> 面向多服务价值链协同的共享制造分布式柔性作业车间多目标调度研究

问题简称：

> MVC-SM-DFJSP，即 Multi-service Value Chain Collaborative Shared Manufacturing Distributed Flexible Job Shop Scheduling Problem

## 1. 论文核心主线

本文以第三方共享制造平台为应用背景。平台连接多条服务价值链、多个制造服务资源单元 SRU 及其内部机器。传统 SM-DFJSP 主要解决“服务类型-SRU-机器”的供需匹配和调度问题，即订单根据制造服务类型选择对应 SRU，再在 SRU 内部完成机器选择与工序排序。

本文在此基础上引入订单级价值链归属，形成“价值链-服务类型-SRU-机器”的层级调度机制。订单首先根据业务归属、客户来源、核心企业或产品族归属于某条价值链；然后根据制造需求划分服务类型；再在链内或跨链的同类型 SRU 中选择加工资源；最后在 SRU 内部进行柔性作业车间调度。本文基础模型中所有 SRU 均开放共享，链内与跨链选择的共同硬约束是服务类型必须匹配。

论文要回答的核心问题是：

> 当多个服务价值链共享第三方制造平台且所有 SRU 均开放共享时，如何在链内资源和跨链资源之间做调度选择，使订单在满足服务类型匹配、机器能力约束和工艺顺序约束的同时，兼顾总成本、最大完工时间和资源负载目标。

需要始终守住四个概念边界：

1. 价值链是订单级业务归属，不是加工类型。
2. 服务类型是订单级制造需求类别，不是价值链。
3. SRU 是共享制造平台上的服务资源单元，可以属于某条价值链；本文基础模型中所有 SRU 均对其他价值链开放共享。
4. 机器是 SRU 内部的执行资源，真正承担工序加工。
5. 无论订单选择链内 SRU 还是跨链 SRU，都必须选择同服务类型的 SRU。

## 2. 研究定位与创新逻辑

### 2.1 原问题基础

原 SM-DFJSP 的主要逻辑是：

```text
订单/作业
  -> 服务类型匹配
  -> SRU 选择
  -> SRU 内部机器选择
  -> 机器工序排序
  -> 总成本、最大完工时间
```

这种逻辑能够体现共享制造中的供需匹配，但不能表达多条服务价值链同时接入平台时的链内协同和跨链协同。

### 2.2 本文扩展逻辑

本文扩展后的逻辑是：

```text
订单池
  -> 订单所属价值链
  -> 订单所属服务类型
  -> 链内/跨链候选 SRU 集合
  -> SRU 分配
  -> SRU 内部机器选择
  -> 机器工序排序
  -> 总成本、最大完工时间、最大 SRU 负载
```

其中，价值链层用于表达业务协同关系，服务类型层用于表达制造能力匹配，SRU 层用于表达资源共享，机器层用于表达具体加工调度。

### 2.3 建议创新点

创新点 1：提出价值链驱动的共享制造调度层级。

在原 SM-DFJSP 的服务类型供需匹配基础上，引入订单级价值链归属，建立“价值链-服务类型-SRU-机器”的层级调度结构，使共享制造调度从单一资源类型匹配扩展到多服务价值链协同。

创新点 2：建立考虑链内与跨链协同的多目标调度模型。

模型同时考虑总成本、最大完工时间和最大 SRU 负载，将跨链固定协同成本、跨链变动成本、运输成本、跨链允许模式和服务类型匹配约束纳入调度决策。

创新点 3：设计面向多价值链协同的改进 EDA-TS 算法。

在原 EDA-TS 基础上，加入价值链感知初始化、价值链约束概率采样、链内/跨链 SRU 替换、跨链回流、瓶颈负载降低邻域和非支配解记忆机制，提高算法在多价值链共享制造调度中的求解能力。

## 4. 应用场景设计

推荐采用航空零部件共享制造或汽车配件共享制造场景。

### 4.1 航空零部件共享制造场景

第三方共享制造平台连接多条航空零部件服务价值链：

- VC1：发动机零部件制造链。
- VC2：机体结构件制造链。
- VC3：维修备件快速响应链。

每条价值链持续发布制造订单。订单可按工序规模或工艺需求划分为不同服务类型，例如短流程件、中长流程件，或车削、铣削、钻削、磨削、检测等类型。平台上存在多个 SRU，每个 SRU 属于某条价值链，并具备一种或多种服务类型能力。本文基础场景假设所有 SRU 均开放共享，但订单无论链内加工还是跨链加工，都只能选择同服务类型 SRU。

当某条价值链内部资源负载较重、交期紧张或成本过高时，平台可以调用其他价值链的同类型 SRU，以换取更短完工时间或更低瓶颈负载，但需要承担跨链固定成本、变动成本、运输成本或服务风险惩罚。

### 4.2 汽车配件共享制造场景

第三方云平台连接多个整车厂、配件代理商、零部件制造商和售后服务商。订单按照整车厂、客户或服务链划分价值链，再根据配件制造需求划分服务类型。平台在链内或跨链同类型 SRU 中选择加工资源，形成多价值链协同制造调度。

如果论文希望更突出柔性作业车间和制造语义，建议选择航空零部件场景；如果希望更贴近多价值链文献，汽车配件场景也可作为背景说明。

## 5. 问题定义

给定：

- 静态到达订单集合 `J`。
- 服务价值链集合 `V`。
- 制造服务类型集合 `T`。
- SRU 集合 `U`。
- 每个 SRU 内部机器集合 `M_u`。
- 每个订单的工序集合 `O_j`。
- 订单与 SRU 间的运输时间、运输成本。
- 链内和跨链协同规则。

需要决定：

1. 每个订单选择链内 SRU 还是跨链 SRU。
2. 每个订单分配给哪个 SRU。
3. 每道工序在所选 SRU 内选择哪台机器。
4. 每台机器上的工序加工顺序。

满足：

1. 订单价值链归属已知。
2. 订单服务类型已知。
3. 订单只能分配给服务类型匹配的 SRU。
4. 所有 SRU 默认开放共享；若实验模式为“不允许跨链”，订单只能选择本价值链内同类型 SRU。
5. 每个订单只由一个 SRU 完成。
6. 每道工序只在所选 SRU 内的一台可行机器上加工。
7. 同一订单内部工序满足工艺顺序。
8. 同一机器同一时刻只能加工一道工序。
9. 加工不可中断。
10. 订单完工时间考虑末端运输时间。

优化目标：

1. 最小化总成本。
2. 最小化最大完工时间。
3. 最小化最大 SRU 负载，或最小化 SRU/价值链负载标准差。

## 6. 研究边界与假设

基础论文建议采用以下边界：

1. 静态订单：所有订单在时刻 0 到达。
2. 订单级价值链固定：价值链归属作为输入，不参与优化。
3. 订单级服务类型固定：服务类型作为输入，不参与优化。
4. 单订单单 SRU：一个订单的所有工序在同一 SRU 内完成。
5. 不考虑工序跨 SRU 转移：避免问题扩展为含转移的 DFJSPT。
6. 不考虑机器故障和动态插单：作为后续扩展。
7. 跨链协同同时有成本惩罚和收益空间。
8. 所有 SRU 均开放共享，开放比例不作为基础模型变量；跨链是否可用由实验模式控制。
9. 先完成二目标工程闭环，再扩展到三目标论文模型。

其中第 7 点很重要。跨链协同不能只增加成本，否则算法会自然避免跨链；必须让部分跨链 SRU 在加工时间、排队负载、瓶颈释放或交期响应方面具备优势，使跨链选择形成真实折中。

## 7. 数学模型

### 7.1 集合

| 符号 | 含义 |
| --- | --- |
| `J` | 订单集合 |
| `V` | 服务价值链集合 |
| `T` | 服务类型集合 |
| `U` | SRU 集合 |
| `M_u` | SRU `u` 内机器集合 |
| `O_j` | 订单 `j` 的工序集合 |
| `A_j` | 订单 `j` 的候选 SRU 集合 |
| `A_j^in` | 订单 `j` 的链内候选 SRU 集合 |
| `A_j^cross` | 订单 `j` 的跨链候选 SRU 集合 |

### 7.2 参数

| 参数 | 含义 |
| --- | --- |
| `vc_j` | 订单 `j` 所属价值链 |
| `type_j` | 订单 `j` 所属服务类型 |
| `vc_u` | SRU `u` 所属价值链 |
| `types_u` | SRU `u` 可服务类型集合 |
| `share_u` | SRU `u` 是否共享开放。基础模型中所有 `share_u = 1`，通常不作为优化约束 |
| `p_{j,o,u,m}` | 订单 `j` 工序 `o` 在 SRU `u` 机器 `m` 上加工时间 |
| `c_{j,o,u,m}` | 对应单位加工成本或加工成本系数 |
| `tt_{j,u}` | 订单 `j` 选择 SRU `u` 的运输时间 |
| `tc_{j,u}` | 订单 `j` 选择 SRU `u` 的运输成本 |
| `cc_{j,u}` | 订单 `j` 跨链选择 SRU `u` 的固定协同成本 |
| `cr_{j,u}` | 订单 `j` 跨链选择 SRU `u` 的变动成本系数 |
| `cap_u` | SRU `u` 的最大负载能力 |

### 7.3 决策变量

| 变量 | 含义 |
| --- | --- |
| `x_{j,u}` | 订单 `j` 是否分配给 SRU `u` |
| `y_{j,o,u,m}` | 工序 `(j,o)` 是否在 SRU `u` 的机器 `m` 上加工 |
| `s_{j,o}` | 工序 `(j,o)` 的开始时间 |
| `e_{j,o}` | 工序 `(j,o)` 的完成时间 |
| `z_{a,b,m}` | 同一机器 `m` 上工序 `a` 与 `b` 的先后关系 |
| `q_{j,u}` | 订单 `j` 是否跨链调用 SRU `u` |

工程实现中，`x/y/z/s/e` 不一定显式建模，可以由编码与解码过程隐式表达。

### 7.4 目标函数

目标 1：最小化总成本。

```text
F1 = PC + TC + CFC
```

其中：

```text
PC  = sum p_{j,o,u,m} * c_{j,o,u,m} * y_{j,o,u,m}
TC  = sum tc_{j,u} * x_{j,u}
CFC = sum cc_{j,u} * q_{j,u}
CVC = 0 (disabled; fixed-only cross-chain penalty).
```

目标 2：最小化最大完工时间。

```text
F2 = max_j (C_j + tt_{j,u})
```

其中 `C_j` 为订单 `j` 最后一道工序完成时间。

目标 3 推荐写法：最小化最大 SRU 负载。

```text
load_u = sum p_{j,o,u,m} * y_{j,o,u,m}
F3 = max_u load_u
```

该目标更准确地表示瓶颈 SRU 负载。论文中建议称为“最大 SRU 负载”或“瓶颈 SRU 负载”，不要直接称为“负载均衡度”。

可选目标 3 扩展：最小化 SRU 负载标准差。

```text
avg_load = sum load_u / |U|
F3_balance = sqrt(sum_u (load_u - avg_load)^2 / |U|)
```

建议主模型采用 `F3 = max_u load_u`，同时在实验指标中报告 `sru_load_std` 作为辅助分析。

### 7.5 约束条件

1. 每个订单只分配给一个 SRU。

```text
sum_u x_{j,u} = 1, for all j
```

2. 订单只能选择候选 SRU。

```text
x_{j,u} = 0, if u not in A_j
```

3. 服务类型必须匹配。该约束对链内选择和跨链选择同时成立。

```text
x_{j,u} = 0, if type_j not in types_u
```

4. 跨链模式约束。

```text
if cross_chain_allowed = 0:
    x_{j,u} = 0, if vc_j != vc_u
```

基础模型中所有 SRU 均开放共享，因此不再用 `open_u` 判断跨链可行性。是否允许跨链由实验模式参数 `cross_chain_allowed` 控制。

5. 跨链变量定义。

```text
q_{j,u} = 1, if x_{j,u} = 1 and vc_j != vc_u
q_{j,u} = 0, otherwise
```

6. 工序只能在已分配 SRU 内加工。

```text
sum_{m in M_u} y_{j,o,u,m} = x_{j,u}
```

7. 工序优先关系。

```text
s_{j,o+1} >= e_{j,o}
```

8. 加工时间关系。

```text
e_{j,o} = s_{j,o} + p_{j,o,u,m}, if y_{j,o,u,m} = 1
```

9. 机器互斥。

同一机器上任意两个工序不能时间重叠，通过排序变量或解码调度规则实现。

10. SRU 负载能力。

```text
load_u <= cap_u
```

如果不希望造成不可行解过多，可将该约束作为软约束，以惩罚项加入评价。

## 8. MVC 实例构造方案

### 8.1 基本思路

以标准 FJSP/MK benchmark 为基础，保留原始订单、工序、机器和加工时间，再人为扩展价值链、服务类型、SRU、运输成本、跨链成本和跨链允许实验模式。本文基础数据中所有 SRU 均开放共享，但订单链内与跨链选择都必须满足服务类型匹配。

构造流程：

```text
基础 FJSP 实例
  -> 读取订单、工序、机器、加工时间
  -> 聚合机器形成 SRU
  -> 分配 SRU 所属价值链
  -> 分配 SRU 服务类型能力
  -> 分配订单所属价值链
  -> 分配订单服务类型
  -> 构造链内/跨链候选 SRU
  -> 构造运输时间与运输成本
  -> 构造跨链固定/变动成本
  -> 构造 SRU 效率、成本和共享属性
  -> 输出 MVC-SM-DFJSP JSON 实例
```

### 8.2 订单价值链生成

推荐采用均衡分配或分块分配：

```text
vc_j = j mod n_value_chains
```

或按订单类别/规模分配，使每条价值链都有足够订单。

论文表述中应说明：价值链代表订单业务归属、客户归属或核心企业归属，而不是制造类型。

### 8.3 服务类型生成

基础阶段可以按工序数划分服务类型：

- `T1`：短流程件。
- `T2`：中长流程件。
- `T3`：复杂流程件，可在扩展数据中使用。

示例规则：

```text
if number_of_operations <= median_ops:
    type_j = T1
else:
    type_j = T2
```

论文中需要说明：这是基于 benchmark 扩展时的“制造需求复杂度类型”，不是业务价值链。

如果后续要增强制造语义，可将类型扩展为车削、铣削、钻削、磨削、检测等。

### 8.4 SRU 构造

SRU 可由机器集合聚合而成。每个 SRU 包含：

- `sru_id`
- `value_chain_id`
- `service_types`
- `machine_ids`
- `share_to_cross_chain`
- `efficiency_factor`
- `cost_factor`
- `capacity`

推荐基础配置：

```text
n_value_chains = 3
n_types = 2
n_srus = 6
```

即每条价值链配置两个 SRU，覆盖两类服务类型，保证每个订单至少存在一个链内可行 SRU。

### 8.5 加工时间与成本调整

基础 FJSP 只给出不同机器的加工时间。扩展为 MVC 后，应根据 SRU 效率和成本系数调整：

```text
p_{j,o,u,m} = ceil(base_p_{j,o,m} * efficiency_factor_u)
c_{j,o,u,m} = base_cost_m * cost_factor_u
```

跨链收益空间可以通过部分跨链同类型 SRU 的较低 `efficiency_factor_u` 体现，即加工更快；跨链代价则通过 `cc/cr/tc` 体现。由于所有 SRU 均开放共享，跨链收益差异主要来自 SRU 效率、成本系数、运输距离和负载状态，而不是开放状态差异。

### 8.6 运输时间与运输成本

可构造价值链距离矩阵：

```text
dist(vc_i, vc_k)
```

链内距离较小，跨链距离较大。订单选择 SRU 后：

```text
tt_{j,u} = base_transport_time * dist(vc_j, vc_u)
tc_{j,u} = base_transport_cost * dist(vc_j, vc_u)
```

为了避免运输成本完全压制跨链收益，建议将运输成本设置为中等强度，并在敏感性分析中调节。

### 8.7 跨链协同成本

如果 `vc_j != vc_u`，则产生：

```text
fixed_cross_cost = cc_{j,u}
variable_cross_cost = cr_{j,u} * processing_cost_{j,u}
```

低、中、高三档成本强度可用于敏感性分析：

```text
low:    cc = 5,  cr = 0.03
medium: cc = 10, cr = 0.06
high:   cc = 20, cr = 0.10
```

具体数值应根据基础实例加工成本量级缩放，保证跨链既不是必选，也不是永远不可选。

### 8.8 跨链允许模式

基础数据中所有 SRU 均开放共享，因此不再设置 SRU 开放比例。实验中通过模式参数控制是否允许跨链：

```text
cross_chain_allowed in {0, 1}
```

- `cross_chain_allowed = 0`：不允许跨链，订单只能选择本价值链内同类型 SRU。
- `cross_chain_allowed = 1`：允许跨链，订单可以选择其他价值链同类型 SRU，并承担跨链成本。

这样可以把“资源是否开放”和“算法是否允许跨链调度”区分开来：前者在本文基础模型中恒为开放，后者作为基准实验变量。

## 10. 编码与解码方案

### 10.1 编码结构

建议采用“五层信息结构、三层进化编码”。

固定信息层：

1. VC 层：订单所属价值链。
2. TYPE 层：订单所属服务类型。

进化编码层：

1. UA：订单到 SRU 的分配。
2. OS：工序排序。
3. MS：机器选择。

这样既体现“价值链-类型-SRU-机器”的论文层级，又避免把订单价值链和服务类型作为无意义的进化变量。

### 10.2 解码顺序

严格按以下顺序解码：

```text
读取订单价值链 vc_j
  -> 读取订单服务类型 type_j
  -> 生成候选 SRU 集 A_j
  -> 由 UA 选择 SRU
  -> 将订单工序放入该 SRU
  -> 由 OS 决定工序调度顺序
  -> 由 MS 决定机器
  -> 计算开始/完成时间
  -> 计算成本、makespan、SRU 负载
```

### 10.3 可行性修复

采样或邻域扰动后执行：

1. 若 UA 选择的 SRU 类型不匹配，替换为同类型候选 SRU。
2. 若当前实验模式不允许跨链但 UA 选择了跨链 SRU，替换为链内同类型 SRU。
3. 若 MS 选择不可加工机器，替换为当前 SRU 内最快或最低成本机器。
4. 若 OS 工序计数错误，按订单工序数补齐或删除多余项。
5. 若 SRU 负载超过硬容量，可优先尝试迁移到同类型低负载候选 SRU，否则加惩罚。

## 11. 本文方法：MVC-EDA-TS

### 11.1 总体流程

```text
输入 MVC 实例与参数
初始化多策略种群
评价种群
建立非支配解记忆池
while 未达到终止条件:
    更新 PMA/PMS/PMM 概率模型
    基于概率模型采样新个体
    执行可行性修复
    执行价值链感知禁忌搜索
    评价新个体
    更新种群
    更新非支配解记忆池
输出 Pareto 解集、指标和调度方案
```

### 11.2 初始化策略

至少包含四类：

1. 成本优先：优先选择加工成本、运输成本和跨链成本综合较低的 SRU。
2. 时间优先：优先选择加工时间和运输时间较短的 SRU。
3. 链内优先：优先选择本价值链 SRU，降低跨链风险。
4. 瓶颈负载优先：优先选择当前累计负载较低的 SRU。

四类策略按比例混合，例如：

```text
25% 成本优先
25% 时间优先
25% 链内优先
25% 负载优先
```

也可加入随机初始化保留多样性。

### 11.3 概率模型

沿用 EDA-TS 的三类概率模型：

- PMA：订单选择 SRU 的概率。
- PMS：工序排序概率。
- PMM：机器选择概率。

PMA 的价值链感知改造：

1. 不可行 SRU 概率固定为 0。
2. 类型不匹配 SRU 概率固定为 0。
3. 若实验模式不允许跨链，则所有跨链 SRU 概率固定为 0。
4. 链内 SRU 可设置较高先验权重。
5. 允许跨链时，跨链同类型 SRU 根据成本、时间和负载收益设置自适应权重。

示例：

```text
score(j,u) =
  w1 * normalized_cost(j,u)
  + w2 * normalized_time(j,u)
  + w3 * normalized_load(u)
  + w4 * cross_chain_penalty(j,u)
```

然后将较低 score 转换为较高采样概率。

### 11.4 禁忌搜索邻域

建议设置五类邻域：

1. 链内 SRU 替换：订单仍在本价值链内，替换为同类型另一个 SRU。
2. 跨链 SRU 替换：订单从本链 SRU 转移到其他价值链同类型 SRU。
3. 跨链回流：跨链订单调回本价值链 SRU。
4. 瓶颈 SRU 释放：从当前最大负载 SRU 中选择订单迁移到低负载候选 SRU。
5. 机器替换与工序重排：在同一 SRU 内改变机器选择或机器上工序顺序。

其中第 2、3、4 类是最能体现本文创新的邻域。

### 11.5 禁忌表与接受准则

禁忌对象可定义为：

```text
(job_id, old_sru, new_sru)
(operation_id, old_machine, new_machine)
(operation_a, operation_b, machine_id)
```

接受准则：

1. 若新解支配当前解，接受。
2. 若新解不被当前解支配且提升多样性，按概率接受。
3. 若新解进入全局非支配解集，即使触发禁忌也允许破禁。
4. 若新解不可行，修复后再评价；无法修复则丢弃或加惩罚。

### 11.6 参数建议

初始建议：

```text
population_size = 80 或 100
generations = 200
elite_ratio = 0.2
sample_ratio = 0.8
tabu_tenure = 7 到 15
local_search_steps = 20 到 50
mutation_rate = 0.05 到 0.15
independent_runs = 20 或 30
```

最终参数需要通过预实验确定，并在论文中固定。

## 12. 基准算法设计

### 12.1 必选基准算法

1. NSGA-II。

经典多目标进化算法，用于证明本文方法相对通用多目标算法的优势。需要适配 MVC 编码、可行性修复和三目标非支配排序。实验中应分别设置 `NSGA-II-cross-off` 与 `NSGA-II-cross-on`，对比不允许跨链和允许跨链两种模式。

2. 原 EDA-TS。

保留原 EDA-TS 的概率建模和禁忌搜索，但不加入价值链感知初始化、跨链邻域和瓶颈释放邻域。用于证明本文改进模块有效。实验中同样设置 `EDA-TS-cross-off` 与 `EDA-TS-cross-on`。

3. MVC-EDA-TS。

本文方法。

本文方法也需要同时报告 `MVC-EDA-TS-cross-off` 与 `MVC-EDA-TS-cross-on`。其中 `cross-off` 用于观察算法在单链约束下的性能，`cross-on` 用于验证跨链协同带来的成本、时间和负载折中。

硕士论文或第一篇论文建议至少完成：

```text
NSGA-II-cross-off vs NSGA-II-cross-on
EDA-TS-cross-off vs EDA-TS-cross-on
MVC-EDA-TS-cross-off vs MVC-EDA-TS-cross-on
```

论文表格中可以按“算法 × 跨链模式”组织，这样既能比较算法，也能比较允许跨链与不允许跨链的机制差异。

### 12.2 推荐扩展基准算法

4. MOEA/D。

分解式多目标进化算法，适合多目标调度问题，可作为三目标场景下的重要基准。

5. NSGA-III。

面向多目标或 many-objective 优化，若第三目标或更多指标扩展明显，可加入。

6. GA-TS 或 H-GA-TS。

混合遗传禁忌搜索算法。DFJSP 文献中常用于平衡全局搜索与局部搜索，可作为和 EDA-TS 同类混合启发式的对照。

7. SPEA2。

经典强度 Pareto 进化算法，可作为补充基线，但不是必须。

### 12.3 基准算法优先级

若时间有限，按以下优先级：

1. NSGA-II。
2. 原 EDA-TS。
3. MVC-EDA-TS。
4. MOEA/D。
5. NSGA-III。
6. GA-TS/H-GA-TS。

不要一开始铺太多算法。优先把三类核心对比跑扎实，再扩展 MOEA/D 或 NSGA-III。

### 13.2 实验二：允许跨链与不允许跨链对比

该实验是必须实验，用于直接回答“跨链协同是否有效”。所有算法都要分别在不允许跨链和允许跨链两种模式下运行。

基础对比模式：

1. Cross-off：不允许跨链，订单只能使用本价值链内同类型 SRU。
2. Cross-on：允许跨链，订单可以使用其他价值链同类型 SRU，并计入跨链固定成本、变动成本和运输成本。

推荐形成以下基准组合：

```text
NSGA-II-cross-off
NSGA-II-cross-on
EDA-TS-cross-off
EDA-TS-cross-on
MVC-EDA-TS-cross-off
MVC-EDA-TS-cross-on
```

可选扩展模式：

1. Type-only-sharing：只考虑服务类型匹配，不考虑价值链归属和跨链成本，用作理想化资源池上界。
2. Full-MVC-3obj：允许跨链，计入跨链成本，并显式优化最大 SRU 负载。

评价内容：

- 跨链协同是否降低 makespan。
- 跨链协同是否增加 total_cost。
- 最大 SRU 负载是否下降。
- 跨链调用比例是否合理。
- 资源利用是否更均衡。

### 13.3 实验三：算法性能对比

数据：

```text
MVC-MK01 至 MVC-MK15
```

每个实例每个算法独立运行 20 或 30 次。

评价指标：

- Hypervolume。
- IGD。
- GD。
- C-metric。
- 非支配解数量。
- 平均运行时间。

- 平均排名。

### 13.4 实验四：消融实验

以 MVC-EDA-TS 为完整算法，设置以下消融版本：

1. 去掉价值链感知初始化。
2. 去掉价值链感知 PMA 先验。
3. 去掉跨链 SRU 替换邻域。
4. 去掉跨链回流邻域。
5. 去掉瓶颈 SRU 释放邻域。
6. 去掉非支配解记忆池。

指标：

- HV 下降幅度。
- IGD 上升幅度。
- 非支配解数量变化。
- 运行时间变化。
- 跨链调用比例变化。

目标：证明每个改进模块确实贡献了 Pareto 质量、搜索效率或协同效果。

### 13.5 实验五：敏感性分析

建议分析六类因素：

1. 跨链固定成本。
2. 跨链变动成本率。
3. 跨链允许模式。
4. 价值链数量。
5. 服务类型数量。
6. 订单规模。

观察问题：

1. 跨链成本升高时，跨链调用比例是否下降。
2. 跨链成本较低时，makespan 是否明显下降。
3. 从不允许跨链切换到允许跨链时，最大 SRU 负载是否下降。
4. 价值链数量增加时，跨链协同收益是否增强。
5. 服务类型数量增加时，候选 SRU 缩小是否导致调度难度上升。
6. 订单规模变大时，MVC-EDA-TS 是否比 NSGA-II 更稳定。

### 13.6 实验六：管理启示分析

从共享制造平台运营角度解释：

1. 跨链共享不是越多越好，存在成本与时间折中。
2. 允许跨链能缓解瓶颈，但可能带来协同成本。
3. 低成本 SRU 可能导致负载集中，需要最大 SRU 负载目标约束。
4. 平台可通过调节跨链成本、跨链准入策略和优先级规则引导资源均衡。

## 15. 论文各章节写作方案

### 第 1 章 绪论

重点写：

## 19. 关键风险与处理

风险 1：跨链只增加成本，不改善时间或负载。

处理：构造实例时必须让部分跨链 SRU 具备效率、排队或瓶颈释放优势，同时设置合理跨链成本。

风险 2：第三目标导致算法和指标复杂度上升。

处理：工程上先跑通二目标，论文模型和正式实验再扩展三目标。保留 `max_sru_load` 和 `sru_load_std` 作为诊断字段。

风险 3：服务类型构造缺少制造语义。

处理：基础 benchmark 用工序数划分服务类型，并在论文中明确其含义是“制造需求复杂度类型”；后续再扩展为工艺类型。

风险 4：算法对原工程改动过大。

处理：新增 MVC 专用 loader、evaluator 和 algorithm，不破坏原 SM-DFJSP 复现代码。

风险 5：实验规模过大。

处理：按“单实例闭环 -> 15 实例主实验 -> 消融 -> 敏感性”的顺序推进。

## 20. 最终建议

本文最稳妥的论文路线是：

1. 理论上明确提出 MVC-SM-DFJSP。
2. 模型上采用“价值链-服务类型-SRU-机器”层级。
3. 目标上以总成本、最大完工时间、最大 SRU 负载为三目标。
4. 数据上从 MK/FJSP benchmark 扩展 MVC 实例。
5. 算法上以 MVC-EDA-TS 为本文方法。
6. 对比上至少使用 NSGA-II 和原 EDA-TS。
7. 实验上必须包含协同机制对比、算法性能对比、消融实验和敏感性分析。

一句话概括：

> 这篇论文不是简单把 FJSP 数据加几个标签，而是要把共享制造中的供需类型匹配、多价值链中的链内/跨链协同，以及 DFJSP 中的 SRU/机器/工序调度统一到一个可求解、可实验、可解释的多目标调度框架中。
