## 三、最重要的建模逻辑：订单先分价值链，再分类型，再选 SRU 和机器

你的论文不要设计成“每道工序划分价值链”，也不要设计成“每道工序重新划分服务类型”。应当采用下面这个层级：

### 1. 第一层：订单/作业分配到价值链

每个订单作业先属于一条价值链。例如：

- 价值链 1：航空发动机零部件链；
- 价值链 2：航空结构件链；
- 价值链 3：航空维修备件链。

或者在汽车配件场景中：

- 价值链 1：整车厂 A 的配件制造服务链；
- 价值链 2：整车厂 B 的配件制造服务链；
- 价值链 3：第三方售后备件服务链。

这里的价值链不是加工类型，而是**业务归属、客户归属、核心企业归属或服务链归属**。多价值链文献中强调第三方平台同时服务多条服务价值链，并具有资源聚合与跨链协同能力，这正好可以作为价值链层的理论来源。`Liu和Sun - 2022 - Multi-service value chains collaboration for repairperson resources selection using a many-objective.pdf`

### 2. 第二层：订单/作业在价值链内部划分服务类型

订单进入某条价值链后，再根据其制造需求划分为某种服务类型。例如：

- 类型 1：车削类；
- 类型 2：铣削类；
- 类型 3：钻削类；
- 类型 4：热处理类；
- 类型 5：检测类。

这个“类型”应当继承共享制造论文中的供需类型思想。共享制造论文明确指出，sharedMfg 的资源组织结构支持基于供需类型的快速检索与匹配，并能将供需匹配和作业车间生产调度集成起来。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf`

### 3. 第三层：在“价值链-类型”对应的 SRU 集合中选择 SRU

一个订单确定了价值链和类型后，只能在满足条件的 SRU 集合中选择。例如：

| 订单 | 价值链 | 服务类型 | 可选 SRU |
|---|---|---|---|
| J1 | VC1 | T1 | SRU1, SRU2 |
| J2 | VC1 | T2 | SRU3 |
| J3 | VC2 | T1 | SRU2, SRU4 |
| J4 | VC3 | T3 | SRU5, SRU6 |

这里可以体现两种协同：

**链内协同**：优先选择本价值链内部的 SRU。  
**跨链协同**：当本链资源成本高、负载重或交期紧时，可以选择其他价值链开放共享的同类型 SRU，但产生跨链协同成本、信任惩罚或服务协同成本。

这与多服务价值链文献中的“链内纵向协同”和“跨链横向协同”思想一致：链内协同是在本服务价值链内部完成服务，跨链横向协同是临时整合其他价值链的资源服务客户。`Liu和Sun - 2022 - Multi-service value chains collaboration for repairperson resources selection using a many-objective.pdf`

### 4. 第四层：在已选 SRU 内部选择机器并排序工序

订单一旦分配到某个 SRU，订单的所有工序都在该 SRU 内部完成；每道工序再根据可加工机器集合选择具体机器，并决定工序加工顺序。这一点继承共享制造论文的 SM-DFJSP 基本逻辑：作业分配到 SRU，工序在 SRU 内部机器上调度，并计算加工成本、运输成本和最大完工时间。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf`

## 四、拟研究问题定义

可以定义为：

**多服务价值链协同共享制造分布式柔性作业车间调度问题，简称 MVC-SM-DFJSP。**

问题描述如下：

给定若干静态到达的订单作业集合、若干服务价值链、若干制造服务类型、若干 SRU 以及 SRU 内部机器集合。每个订单作业首先归属于一条价值链，并具有一种制造服务类型；每个 SRU 也隶属于某条价值链，并具备一种或多种服务类型。平台需要在满足价值链归属、供需类型匹配、SRU 能力约束、机器能力约束和工序优先约束的前提下，决定：

1. 每个订单是否采用链内 SRU 或跨链 SRU；
2. 每个订单分配到哪个 SRU；
3. 每道工序选择 SRU 内的哪台机器；
4. 每台机器上的工序加工顺序；
5. 最终形成兼顾成本、完工时间和跨链协同效果的调度方案。

### 场景版本 1：航空零部件共享制造

第三方共享制造平台连接多条航空零部件价值链：

- VC1：发动机零部件制造链；
- VC2：机体结构件制造链；
- VC3：维修备件快速响应链。

## 七、数学模型构思

### 1. 集合

可以设置：

- 订单作业集合：所有静态订单；
- 价值链集合：多条服务价值链；
- 服务类型集合：不同制造服务类型；
- SRU 集合：共享制造平台上的服务资源单元；
- 机器集合：每个 SRU 内部的机器；
- 工序集合：每个订单包含若干工序。

### 2. 参数

需要包括：

- 每个订单所属价值链；
- 每个订单所属服务类型；
- 每个 SRU 所属价值链；
- 每个 SRU 能提供的服务类型；
- 每道工序在不同机器上的加工时间；
- 每道工序在不同机器上的单位加工成本；
- 订单客户位置与 SRU 之间的运输时间；
- 订单客户位置与 SRU 之间的运输成本；
- 链内协同成本；
- 跨链协同成本；
- 跨链服务惩罚系数；
- SRU 最大负载能力；
- 机器可加工能力矩阵。

### 3. 决策变量

建议分为五类：

第一类：价值链归属变量。  
表示订单属于哪条价值链。静态研究中这个变量可以是已知参数，也可以作为平台分配决策。如果你要符合“订单作业先划分价值链”，建议在基础模型中将其作为已知参数；在扩展模型中再考虑平台可调整价值链归属。

第二类：类型归属变量。  
表示订单属于哪种制造服务类型。基础模型中也建议作为已知参数。

第三类：SRU 分配变量。  
表示订单是否分配到某个 SRU。

第四类：机器选择变量。  
表示某订单某工序是否选择某 SRU 内某机器加工。

第五类：排序变量。  
表示同一机器上两个工序的先后顺序。

### 4. 主要约束

模型至少要有以下约束：

**约束 1：每个订单只能属于一条价值链。**  
静态模型中可直接给定。

**约束 2：每个订单只能属于一种服务类型。**  
这里是订单级类型，不是工序级类型。这样可以保证“先价值链、再类型”的层级清晰。

**约束 3：订单只能分配给服务类型匹配的 SRU。**  
这是共享制造论文中供需一致性的扩展。共享制造论文强调不同作业有不同需求、不同 SRU 提供不同服务，因此需要实现供需一致性。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf`

**约束 4：订单可选择链内 SRU，也可选择跨链 SRU，但跨链 SRU 必须开放共享且类型匹配。**  
这是多价值链协同的关键约束。

**约束 5：每个订单只能分配给一个 SRU。**  
先做静态基础研究时，不建议允许同一订单跨 SRU 加工，否则会变成含转移的 DFJSPT，模型复杂度大幅增加。含转移文献研究的是工序可以在不同工厂间转移，并考虑最大完工时间、最大工厂负载和总能耗等目标。`一种高效的分布式柔性作业车间调度问题（含转移）模因算法.pdf` 你的第一篇或硕士论文基础模型不必直接引入工序跨 SRU 转移。

**约束 6：每道工序只能选择该 SRU 内一台可加工机器。**

**约束 7：同一订单内部工序满足工艺优先顺序。**

**约束 8：同一机器同一时刻只能加工一道工序。**

**约束 9：不可中断加工。**

**约束 10：完工后考虑从 SRU 到客户或需求节点的运输时间与运输成本。**  
共享制造论文中总成本包含生产成本和 SRU 到客户之间的运输成本，最大完工时间也考虑了工件完工后运输到客户的时间。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf`

## 八、目标函数设置：先做静态订单，建议用 3 个目标

原共享制造论文使用两个目标：**总成本**和**最大完工时间**。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf` 你的论文要体现多价值链，建议在这两个目标基础上增加一个“跨链协同/链间负载均衡”目标。

### 目标 1：最小化总成本

总成本包括：

1. 加工成本；
2. 运输成本；
3. 跨链协同成本；
4. 跨链服务惩罚成本。

也就是：

> 总成本 = 所有工序加工成本 + 所有订单运输成本 + 跨链调用成本 + 跨链协同惩罚成本。

其中跨链协同成本是你的多价值链特色。如果订单属于 VC1，但选择了 VC2 下的 SRU，就产生跨链成本；如果订单选择本链 SRU，则没有跨链成本或成本较低。

### 目标 2：最小化最大完工时间

最大完工时间应包括：

> 订单最后一道工序完成时间 + 从所选 SRU 到客户位置的运输时间。

这样继承共享制造论文的 makespan 计算思想。

### 目标 3：最小化链间资源失衡或最大 SRU 负载

这个目标用于体现“多价值链协同不是简单跨链调用，而是为了提高整体资源利用率”。

可以有两个写法：

**写法 A：最小化最大 SRU 负载。**

即让最忙 SRU 的总加工时间尽量小，避免所有订单集中到某一个低成本 SRU。

**写法 B：最小化价值链负载偏差。**

即让不同价值链承担的加工负载更均衡，避免某条价值链资源过载，另一条价值链资源闲置。

多服务价值链资源选择文献指出，多价值链协同可以提升资源利用率，并使多链协同下的资源满足率高于单链协同。`Liu和Sun - 2022 - Multi-service value chains collaboration for repairperson resources selection using a many-objective.pdf` 因此，在调度模型中加入资源均衡或利用率目标是合理的。

## 九、算法构思

### 1. 基础算法：继承共享制造论文的 EDA-TS

共享制造论文采用 EDA-TS，包含估计分布算法和禁忌搜索，并引入多种群策略和非支配解记忆机制。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf` 你的论文可以在此基础上改进为：

**面向多价值链层级编码的多目标 EDA-TS 算法。**

### 2. 编码设计

原共享制造论文使用四层编码：

- UA：SRU assignment；
- OS：operation sequencing；
- OP：operation processing；
- MS：machine selection。`基于共享制造的分布式柔性作业车间调度与供需匹配.pdf`

你的论文可以扩展为五层或保留四层但改造 UA。

建议使用五层编码：

1. **VC 层**：价值链层，表示订单所属价值链。静态模型中可固定，不参与进化；
2. **TYPE 层**：类型层，表示订单所属制造服务类型。静态模型中可固定，不参与进化；
3. **UA 层**：SRU 分配层，表示每个订单选择哪个 SRU；
4. **OS 层**：工序排序层，表示各类型订单的工序排序；
5. **MS 层**：机器选择层，表示每道工序选择 SRU 内哪台机器。

如果你想降低算法复杂度，也可以把 VC 层和 TYPE 层作为输入参数，不编码；只在解码时用于限制 UA 层的可选 SRU 集合。

### 3. 解码过程

解码时严格按照以下顺序：

第一步，读取订单所属价值链。  
第二步，读取订单所属服务类型。  
第三步，根据“价值链-类型-SRU开放状态”生成候选 SRU 集合。  
第四步，依据 UA 层确定订单分配的 SRU。  
第五步，将订单工序加入该 SRU 的待加工队列。  
第六步，依据 OS 和 MS 层确定工序在机器上的开始时间、结束时间。  
第七步，计算成本、完工时间、链间负载等目标值。

### 4. 初始化策略

可以设计四类初始个体：

1. 成本优先初始化：优先选择单位加工成本低的 SRU 和机器；
2. 时间优先初始化：优先选择加工时间短、运输时间短的 SRU；
3. 链内优先初始化：优先选择本价值链 SRU；
4. 负载均衡初始化：优先选择当前负载较低的 SRU。

这样既继承共享制造论文中的调度规则思想，也突出多价值链协同。

### 5. 邻域搜索设计

禁忌搜索或局部搜索可以设计三类邻域：

**邻域 1：链内 SRU 替换。**  
订单仍在原价值链内，只替换为同类型的另一个 SRU。

**邻域 2：跨链 SRU 替换。**  
订单从本链 SRU 转移到其他价值链开放共享的同类型 SRU，或从跨链 SRU 调回本链 SRU。

**邻域 3：机器替换与工序重排。**  
在同一 SRU 内改变某工序的机器选择，或交换同机器上的工序顺序。

这比原共享制造论文更有创新性，因为它引入了“跨链资源协同邻域”。

## 十二、可以提炼的创新点

建议写成 3 个创新点：

### 创新点 1：提出价值链驱动的共享制造调度层级

不同于原 SM-DFJSP 只考虑供需类型匹配，本文进一步引入价值链归属，建立“价值链-服务类型-SRU-机器”的层级调度结构，使订单先在业务层归属价值链，再在制造层完成类型匹配和资源调度。

### 创新点 2：建立考虑链内/跨链协同的多目标调度模型

模型同时考虑总成本、最大完工时间和链间资源负载均衡，并将跨链协同成本、跨链服务惩罚和 SRU 开放共享机制纳入约束与目标。

### 创新点 3：设计面向多价值链协同的改进 EDA-TS 算法

在原 EDA-TS 的基础上，设计价值链感知编码、链内/跨链 SRU 替换邻域、负载均衡初始化和非支配解记忆机制，提高算法对多价值链共享制造调度问题的求解能力。
